# Next.js + Supabase: збереження та Realtime

## Загальний принцип

У статичного Next.js-сайту на GitHub Pages немає власного backend-сервера.
Браузер напряму працює з Supabase через publishable key:

```text
Next.js у браузері
        ↓
Supabase JavaScript Client
        ↓
PostgREST API → PostgreSQL
        ↓
Supabase Realtime → інші відкриті браузери
```

Publishable key і URL потрапляють у JavaScript bundle. Це нормально: доступ до
таблиць контролює не прихованість ключа, а Row Level Security — RLS.

Ніколи не додавайте у frontend:

- `service_role`;
- secret key;
- пароль бази даних.

## Навіщо потрібен SQL migration

Frontend може читати й змінювати дозволені дані, але не повинен створювати
таблиці. Структура бази створюється окремим SQL migration:

```sql
create table public.notes (
  id bigint generated by default as identity primary key,
  text_content text not null,
  updated_at timestamptz not null default now()
);

alter table public.notes enable row level security;

create policy "Public notes access"
on public.notes
for all
to anon
using (true)
with check (true);
```

Migration виконують один раз у Supabase SQL Editor. Після цього frontend може
працювати з таблицею через publishable key.

## Простий приклад

### Створення клієнта

```ts
import { createClient } from "@supabase/supabase-js";

export const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!,
);
```

### Читання

```ts
const { data, error } = await supabase
  .from("notes")
  .select("*")
  .order("id");

if (error) throw error;
console.log(data);
```

### Створення

```ts
const { error } = await supabase
  .from("notes")
  .insert({ text_content: "Перша нотатка" });

if (error) throw error;
```

### Оновлення

```ts
const { error } = await supabase
  .from("notes")
  .update({ text_content: "Оновлений текст" })
  .eq("id", 1);

if (error) throw error;
```

## Більший приклад: autosave із debounce та retry

Не варто надсилати запит на кожен символ textarea. Краще зачекати кілька секунд
після останньої зміни:

```ts
type SaveStatus = "idle" | "pending" | "saving" | "saved" | "error";

let timer: ReturnType<typeof setTimeout> | undefined;

function scheduleSave(id: number, notes: string) {
  setStatus("pending");
  clearTimeout(timer);

  timer = setTimeout(() => {
    void saveWithRetry(id, notes);
  }, 3000);
}

async function saveWithRetry(id: number, notes: string) {
  for (let attempt = 1; attempt <= 3; attempt += 1) {
    setStatus("saving");

    const { error } = await supabase
      .from("roadmap_items")
      .upsert({ id, notes }, { onConflict: "id" });

    if (!error) {
      setStatus("saved");
      return;
    }
  }

  setStatus("error");
}
```

У реальному React-компоненті timer краще тримати в `useRef`, а під час unmount
викликати `clearTimeout`. Для кількох карток використовується `Map<id, timer>`,
щоб кожна картка мала незалежний debounce.

Стани можна показувати маленьким індикатором:

```text
жовтий  pending  — є незбережені зміни
фіолет saving   — запит виконується
зелений saved    — база підтвердила збереження
червоний error   — три спроби не вдалися, доступний ручний retry
```

## Realtime між браузерами

Спочатку таблицю потрібно додати до Realtime publication:

```sql
alter publication supabase_realtime
add table public.roadmap_items;
```

Після цього клієнт може підписатися на зміни:

```ts
const channel = supabase
  .channel("roadmap-live")
  .on(
    "postgres_changes",
    {
      event: "*",
      schema: "public",
      table: "roadmap_items",
    },
    (payload) => {
      if (payload.eventType !== "DELETE") {
        updateLocalState(payload.new);
      }
    },
  )
  .subscribe();

// Cleanup при закритті компонента:
await supabase.removeChannel(channel);
```

Якщо один браузер змінює картку, Supabase:

1. записує update у PostgreSQL;
2. створює Realtime-подію;
3. надсилає її всім підписаним браузерам;
4. кожен браузер оновлює свій React state.

## Захист незбережених змін

Realtime-подія не повинна перезаписати текст, який користувач ще друкує.
Тому перед застосуванням зовнішньої події треба перевірити локальний save status:

```ts
const status = saveStates[itemId];

if (
  status !== "pending" &&
  status !== "saving" &&
  status !== "error"
) {
  applyRealtimeUpdate(row);
}
```

Після успішного збереження відповідь бази містить остаточне значення та
`updated_at`. Саме її слід вважати підтвердженим станом.

## Важливе про безпеку

Поточний roadmap не має входу користувачів, тому RLS дозволяє анонімне
редагування. Будь-хто, хто відкриє сайт, технічно може змінювати таблицю.

Для приватного робочого інструмента наступним кроком має бути Supabase Auth.
Після додавання входу RLS-політики потрібно обмежити роллю `authenticated` або
конкретними user ID.
